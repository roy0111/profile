What is the difference between Future and CompletableFuture in Java? How do you compose them?

Future: Introduced in Java 5, Future represents the result of an asynchronous computation. However, it has no callback capabilities; you must block the executing thread by calling get() or poll in a loop using isDone() to retrieve results.

CompletableFuture: Introduced in Java 8, CompletableFuture implements Future and CompletionStage. It provides a rich set of non-blocking monadic methods, enabling you to build asynchronous pipelines with callbacks and reactive mappings.

Pipeline Composition: You can chain asynchronous tasks using thenApply (transform value), thenCompose (flatmap another future), and thenCombine (join two independent futures). It also integrates exception handlers like exceptionally to resolve pipeline failures gracefully.

Thread Management: If no custom executor is supplied, actions run in the common ForkJoinPool thread context. CompletableFuture methods ending in "Async" allow offloading tasks to specific thread pools to prevent blocking core app resources.

CompletableFuturePipeline.java 
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

public class CompletableFuturePipeline {
    public static void main(String[] args) throws Exception {
        ExecutorService customPool = Executors.newFixedThreadPool(4);

        // Chain async operations with exception handling and timeouts
        CompletableFuture.supplyAsync(() -> {
            System.out.println("Fetching user on: " + Thread.currentThread().getName());
            return "User-99";
        }, customPool)
        .thenApplyAsync(userId -> {
            System.out.println("Processing profile on: " + Thread.currentThread().getName());
            return userId + " (Senior Developer)";
        }, customPool)
        .thenCompose(profile -> CompletableFuture.supplyAsync(() -> profile + " | Score: 98"))
        .exceptionally(ex -> {
            System.err.println("Pipeline failed: " + ex.getMessage());
            return "Fallback Profile";
        })
        .thenAccept(result -> System.out.println("Final Result: " + result));

        // Sleep main thread briefly to allow async pipeline execution
        Thread.sleep(1000);
        customPool.shutdown();
    }
}

Explain the transition from javax to jakarta package namespace in modern Enterprise Java. What are the key implications?

Migration History: Oracle transferred control of Java EE to the Eclipse Foundation in 2017. As part of this transition, Oracle retained ownership of the "java" and "javax" trademarks, meaning the Eclipse Foundation was forbidden from modifying APIs under the javax.* namespace.

Namespace Change: To evolve the specifications, Eclipse rebranded Java EE to Jakarta EE. Starting with Jakarta EE 9, all enterprise APIs transitioned from the javax.* package namespace to the jakarta.* namespace (e.g., servlet, JPA, validation).

Key Implications: Modern frameworks (Spring Boot 3.x, Hibernate 6.x) depend on Jakarta EE 10. Existing applications migrating to these versions must update all imports (e.g., from javax.persistence.* to jakarta.persistence.*) or use build tool transformation plugins.

Compatibility: Legacy libraries using javax.* will crash at runtime with NoClassDefFoundError in modern application servers (like Tomcat 10+ or WildFly 27+) unless an active class bytecode transformation layer is configured.

What is a Lambda Expression in Java? Compare general Functional Interfaces with the java.util.function.Predicate interface.

Lambda Expression: A lambda expression is a short block of code that accepts parameters and returns a value. It provides a clean, expressive way to represent a functional interface inline without creating an anonymous class boilerplate, improving readability in stream operations.

Functional Interface: A functional interface is any interface that contains exactly one abstract method (Single Abstract Method - SAM). It can contain multiple default or static methods, and is marked with the optional @FunctionalInterface annotation to enforce compilation checks.

Predicate Interface: Predicate<T> is a built-in functional interface defined in java.util.function. It declares a single abstract method boolean test(T t), designed to evaluate an input object against a condition and return a boolean.

Comparison: While a general functional interface can return any type or throw custom checked exceptions, Predicate is strictly a boolean-valued function. Predicates can also be chained together using default methods like and(), or(), and negate().

Explain the usage of static, final, and default keywords in Java. What is the significance of the main method's signature?

Static: The static keyword indicates that a member (variable, method, or nested class) belongs to the class itself rather than to instances of the class. Static members are initialized once when the class is loaded into memory, shared across all instances, and can be accessed without creating an object.

Final: The final keyword enforces immutability and restricts inheritance. (1) A final variable cannot be reassigned once initialized. (2) A final method cannot be overridden by subclasses. (3) A final class cannot be subclassed, which is crucial for creating immutable classes like String.

Default: Introduced in Java 8, default methods allow interfaces to declare method implementations. This enables developers to add new methods to existing interfaces without breaking the classes that implement them, maintaining backward compatibility.

Main Method Signature: The standard entry point public static void main(String[] args) is designed for JVM startup: public permits the JVM to invoke it from outside the package, static allows invocation without instantiating the class, void specifies that it returns no value, and String[] args accepts runtime command-line arguments.

What are the major features introduced in Java 17? Explain Records, Sealed Classes, Pattern Matching, and Switch Expressions.

Java 17 (released in September 2021) is a Long-Term Support (LTS) release containing several major language improvements:

  • Records: Concise immutable data carriers. The compiler automatically generates the constructor, getters (without `get` prefix), `equals()`, `hashCode()`, and `toString()`. Fields are final and records cannot extend other classes.
  • Sealed Classes & Interfaces: Allow class developers to restrict which classes/interfaces can extend or implement them. Useful for domain modeling where hierarchies are closed and known. Subclasses must be declared as `final`, `sealed`, or `non-sealed`.
  • Pattern Matching for instanceof: Eliminates the boilerplate of explicit casting. You declare the binding variable directly in the `instanceof` check.
  • Switch Expressions: Return values directly from switch blocks, use arrow syntax (`->`), and guarantee compile-time exhaustiveness (especially when matching sealed hierarchies).

How do you design non-blocking asynchronous pipelines using CompletableFuture? Show how to handle pipeline errors and enforce timeouts.

CompletableFuture (introduced in Java 8) implements the Future and CompletionStage interfaces, enabling non-blocking declarative task chaining. Unlike classic Futures, it supports callbacks (e.g., thenApply, thenAccept, thenCompose) that execute asynchronously when upstream computations complete, avoiding thread blocking (`.get()`).

Pipeline Error Handling: Exceptions that occur in asynchronous stages propagate down the pipeline. You can intercept and recover from exceptions using:

  • exceptionally(): Acts as a catch-block; returns a default fallback value of the same type.
  • handle(): Receives both the result and the exception, allowing you to transform either into a new output.
  • whenComplete(): Serves as a finally-block, executing callbacks without altering the pipeline value.

Timeout Enforcement (Java 9+): You can use `.orTimeout(timeout, unit)` to throw a TimeoutException if a stage does not complete within limits, or `.completeOnTimeout(defaultValue, timeout, unit)` to return a default value on timeout.

AsyncOrchestrator.java 
import java.util.concurrent.CompletableFuture;
import java.util.concurrent.TimeUnit;

public class AsyncOrchestrator {
    public static void main(String[] args) {
        CompletableFuture.supplyAsync(() -> {
            // Simulate fetching user details from remote microservice
            sleep(100);
            return "User: Birat";
        })
        .thenCompose(user -> CompletableFuture.supplyAsync(() -> {
            // Fetch configuration/roles based on user details
            sleep(150);
            return user + " | Role: Admin";
        }))
        // Enforce a strict timeout of 500ms on the pipeline
        .orTimeout(500, TimeUnit.MILLISECONDS)
        // Handle success/failure smoothly
        .handle((result, ex) -> {
            if (ex != null) {
                System.err.println("Pipeline failed: " + ex.getMessage());
                return "Fallback User Config (Default)";
            }
            return result;
        })
        .thenAccept(finalResult -> System.out.println("Result: " + finalResult))
        .join(); // Block main thread to allow async threads to finish output
    }

    private static void sleep(long ms) {
        try { Thread.sleep(ms); } catch (InterruptedException e) { Thread.currentThread().interrupt(); }
    }
}

Explain the evaluation lifecycle, performance, and memory implications of Java Streams vs standard loops. When do parallel streams degrade throughput?

Evaluation Lifecycle: Java Streams are built on a pipeline architecture comprised of a source, zero or more intermediate operations, and exactly one terminal operation. Operations are divided into lazy (e.g., filter, map, flatMap) and eager (terminal operations like collect, findFirst, forEach). Lazy operations do not perform any processing until the terminal operation is executed. During execution, intermediate operations are fused together in a single pass over the elements (loop fusion), minimizing traversal overhead.

Architectural Tip: ForkJoinPool Sharing

By default, all parallel streams execute using the JVM-wide common ForkJoinPool (ForkJoinPool.commonPool()). If a parallel stream performs blocking I/O (e.g., external REST API calls or DB queries), it will saturate the common pool, blocking threads across the entire application. To safeguard production systems, wrap blocking stream operations inside a custom, isolated ForkJoinPool context.

Memory & Performance: Streams incur Object Allocation overhead due to pipeline wrapper instantiation, lambda context capture, and primitive boxing. If processing million-record collections of primitives (like integers), using Stream<Integer> creates massive boxing overhead compared to primitive loops. Use specialized primitive streams like IntStream, LongStream, or DoubleStream to eliminate boxing.

Parallel Streams Degrade Performance When:
  • Small Data Sizes: The overhead of partitioning the collection, context-switching worker threads, and merging sub-results exceeds the cost of single-threaded linear iterations.
  • Splitting Costs are High: Streams backed by LinkedList split terribly (O(N) partition time) compared to arrays or ArrayList (O(1) split time).
  • Stateful Operations: Operators like sorted(), distinct(), or limit() require global coordination, rendering parallelization highly inefficient.

CustomForkJoinParallelStream.java 
import java.util.List;
import java.util.concurrent.ForkJoinPool;
import java.util.stream.Collectors;
import java.util.stream.IntStream;

public class CustomForkJoinParallelStream {
    public static void main(String[] args) throws Exception {
        List<Integer> dataset = IntStream.range(1, 100000).boxed().collect(Collectors.toList());

        // Create an isolated ForkJoinPool with custom parallelism
        ForkJoinPool customPool = new ForkJoinPool(4); 
        
        try {
            // Executing stream execution inside custom pool thread context
            List<Double> processed = customPool.submit(() -> 
                dataset.parallelStream()
                       .filter(n -> n % 2 == 0)
                       .map(Math::sqrt)
                       .collect(Collectors.toList())
            ).get(); // Synchronous block for execution
            
            System.out.println("Processed size: " + processed.size());
        } finally {
            customPool.shutdown(); // Ensure resource cleanup
        }
    }
}

What is a functional interface?

A Functional Interface is an interface that has exactly one abstract method (SAM - Single Abstract Method). It can contain any number of default or static methods. It is marked with the optional `@FunctionalInterface` annotation, which causes the compiler to verify its structure.

Java 8 provides standard functional interfaces in the java.util.function package:

  • Predicate<T>: Takes one argument of type `T` and returns a `boolean` (method: `test(T)`).
  • Function<T, R>: Takes one argument of type `T` and returns a result of type `R` (method: `apply(T)`).
  • Consumer<T>: Takes one argument of type `T` and returns `void` (method: `accept(T)`).
  • Supplier<T>: Takes no arguments and returns a result of type `T` (method: `get()`).

Optional Usage: Optional<T> is a container object used to represent the presence or absence of a value, helping prevent NullPointerException. Best practices: Avoid using Optional as field variables or method parameter types (intended solely for method return types). Use functional pipelines like `map()`, `filter()`, and `flatMap()`, or fallback operations like `orElse()` and `orElseGet()`, instead of nesting `if (opt.isPresent())` blocks.

FunctionalInterfaceAndOptional.java 
import java.util.Optional;
import java.util.function.Function;
import java.util.function.Predicate;

public class FunctionalInterfaceAndOptional {
    public static void main(String[] args) {
        Predicate<String> isNumeric = str -> str != null && str.matches("\\d+");
        Function<String, Integer> toInt = Integer::parseInt;

        String rawInput = "1024";

        // Optional pipeline mapping and filtering

            .filter(isNumeric)
            .map(toInt);

        // Print value if present, otherwise fallback
        System.out.println("Mapped Integer: " + result.orElse(-1));
    }
}

How many threads can a JVM create? What configuration parameters and system limits restrict thread allocation?

Thread Allocation Limits: The maximum thread limit is not a fixed JVM setting; it is determined by OS limitations and available hardware resources.

Operating System Limits: Key constraints include: (1) OS maximum user processes (ulimit -u), (2) kernel max threads (/proc/sys/kernel/threads-max), and (3) available physical RAM.

Stack Memory Constraint: Each thread reserves a thread stack size (specified via -Xss, usually 1MB). If you allocate too many threads, physical RAM gets depleted.

Out of Memory: Exhausting these limits throws java.lang.OutOfMemoryError: unable to create new native thread. Use bounded ThreadPools to prevent crash-on-spike.

BoundedThreadPoolConfig.java 
import java.util.concurrent.*;

public class BoundedThreadPoolConfig {
    public static void main(String[] args) {
        // Enforce thread boundaries to prevent native memory exhaustion
        int corePoolSize = 10;
        int maxPoolSize = 50;
        long keepAliveTime = 60L;
        
        // Use a bounded queue to prevent Infinite Memory Accumulation (OOM)
        BlockingQueue<Runnable> queue = new LinkedBlockingQueue<>(500);

        // Custom ThreadFactory to name threads for debugging
        ThreadFactory threadFactory = new ThreadFactory() {
            private int count = 1;
            @Override
            public Thread newThread(Runnable r) {
                return new Thread(r, "AppWorker-" + count++);
            }
        };

        // AbortPolicy rejects tasks when thread limits and queue boundaries are saturated
        RejectedExecutionHandler handler = new ThreadPoolExecutor.AbortPolicy();

        ExecutorService executor = new ThreadPoolExecutor(
            corePoolSize,
            maxPoolSize,
            keepAliveTime,
            TimeUnit.SECONDS,
            queue,
            threadFactory,
            handler
        );

        System.out.println("Bounded thread pool initialized safely.");
        executor.shutdown();
    }
}

How does Garbage Collection reclaim memory in the JVM? Compare G1GC vs ZGC collectors and their production configuration.

Generational Heap: JVM Garbage Collection organizes the heap into Young (Eden, Survivor spaces) and Old generations. Objects are created in Eden, promoted to survivor spaces, and eventually moved to the Old generation if they persist.

G1GC (Garbage-First): Designed for multi-gigabyte heaps, G1GC divides the heap into thousands of virtual regions. It prioritizes regions containing the most garbage, compacting space concurrently while tracking a configurable target pause time.

ZGC (Z Garbage Collector): A low-latency, scalable collector. ZGC performs memory reclamation concurrently with application threads, using colored pointers and load barriers to resolve memory state issues.

GC Pause Times: ZGC restricts pause times to under a millisecond regardless of heap size, making it suitable for modern enterprise microservices running on huge heaps (GBs to TBs).

Compare Heap vs Stack memory in Java. Where are objects and class definitions saved? Discuss primitive vs wrapper classes.

Stack Memory: Stack memory stores thread execution frames, local primitive variables, and references to objects on the heap. It is allocated per thread, operates in LIFO order, is extremely fast, and requires no garbage collection.

Heap Memory: Heap memory stores all instantiated objects and arrays. It is shared across all application threads, has a dynamic size, and requires Garbage Collection (GC) overhead to reclaim memory.

Metaspace: Introduced in Java 8 to replace PermGen, Metaspace resides in native off-heap memory. It stores class definitions, method metadata, static variables, and the runtime constant pool.

Primitives vs Wrappers: Primitive types (e.g. int) store values directly on the stack (unless they are fields inside a heap-allocated object). Wrapper types (e.g. Integer) are full objects allocated on the heap, introducing boxing and unboxing performance overhead.

How do hash collisions occur in Java HashMaps? How are they resolved internally in Java 8+?

Hash Collisions: A collision occurs when two distinct keys yield the same hash code and map to the exact same bucket index in the underlying node array. The HashMap must store both entries in the same bucket.

Chaining: Historically, HashMaps resolved collisions using linked lists. Multiple colliding nodes are chained together in a singly linked list structure inside the bucket, degrading lookup performance to O(N).

Treeification: In Java 8+, if a bucket's node count exceeds the threshold TREEIFY_THRESHOLD = 8, and the total table capacity is at least 64, the linked list is converted into a balanced Red-Black Tree.

Time Complexity: Treeification optimizes lookup performance, changing worst-case time complexity from O(N) to O(log N). This protects applications against Hash Collision Denial of Service (DoS) attacks.

import java.util.HashMap;

    }
}

What makes a collection thread-safe? Compare Fail-Fast vs Fail-Safe iterators with real-world collection examples.

Thread Safety: A collection is thread-safe if it guarantees structural and data consistency under concurrent modifications by multiple threads. This is achieved via mutual exclusion locks, lock-striping, or copy-on-write mechanisms.

Fail-Fast Iterators: Iterators from standard collections (like ArrayList or HashMap) are fail-fast. If the collection is modified structurally during iteration, the iterator immediately throws a ConcurrentModificationException.

ModCount Mechanism: Fail-fast iterators track modification states using a volatile modCount counter. At each step, the iterator checks if the counter has changed; if a mismatch is detected, it triggers a fast-fail crash.

Fail-Safe Iterators: Iterators from concurrent collections (like CopyOnWriteArrayList or ConcurrentHashMap) are fail-safe. They operate on a copy or snapshot of the collection, allowing concurrent updates without raising exceptions.

IteratorComparison.java 
import java.util.ArrayList;
import java.util.List;

        }

    }
}

Compare ArrayList, LinkedList, and Vector in Java. How do they perform under different memory and concurrency workloads?

ArrayList: Backed by a dynamically resizing array. It provides O(1) time complexity for random access index lookup. However, inserting or deleting elements from arbitrary positions requires O(N) time because subsequent elements must be shifted in memory.

LinkedList: Structured as a doubly-linked list. It provides O(1) insertions or deletions at endpoints. Random access, however, requires O(N) traversal from the head or tail to find the target index, and it consumes more memory per node to store address pointers.

Vector: A legacy, thread-safe dynamic array. It synchronizes every operation using monitor locks. This ensures thread safety but introduces significant synchronization overhead, making it inefficient for single-threaded or modern concurrent pipelines.

Selection Strategy: Use ArrayList as the default list. Switch to LinkedList only if your workload relies heavily on head/tail insertions (like queues or deques). Under concurrent conditions, use a lock-free CopyOnWriteArrayList or wrap lists using Collections.synchronizedList.

Compare Comparable vs Comparator interfaces. When should each be used?

Comparable Interface: The Comparable interface defines the natural ordering of objects. It requires implementing the compareTo(T o) method directly within the target class, which can only support a single, default sorting strategy.

Comparator Interface: The Comparator interface represents an external, pluggable sorting logic. It implements the compare(T o1, T o2) method, allowing you to define multiple independent sorting strategies for the same class.

Use Cases: Use Comparable to establish a default sorting behavior (e.g. sorting Employee objects by ID). Use Comparator when you need to sort by alternate criteria (e.g. by salary, name, or hire date) or when you cannot modify the source code of the target class.

Modern API Support: Java 8 enhanced Comparator with static helper methods like comparing() and thenComparing(), which integrate cleanly with lambda syntax to build complex sorting chains.

What are Marker Interfaces in Java? Discuss how Serialization works, including its security and performance pitfalls.

Marker Interfaces: A marker interface (also called tagged interface) is an interface containing no declared methods or constants (e.g. Serializable, Cloneable). It acts as a metadata tag, instructing the compiler or JVM to apply specific runtime behavior.

Serialization Mechanics: Object serialization converts an object's complete instance state into a portable binary byte stream. This stream can be written to disk, stored in memory, or sent over networks. Deserialization reverses this, reconstructing the object state.

Security Pitfalls: Deserialization is highly vulnerable. If an application deserializes untrusted streams, attackers can inject malicious class states (gadget chains) that trigger remote code execution (RCE) inside the JVM.

Performance Overhead: Default Java serialization is slow and bloats file sizes because it embeds complete class metadata, graph structures, and fully qualified class path descriptors in the byte stream. Modern systems prefer lighter alternatives like JSON or Protocol Buffers.

What is the volatile keyword in Java? Compare transient vs volatile variables.

Volatile: The volatile keyword prevents threads from caching variables in CPU registers, forcing all reads and writes directly to the shared Main Memory. This guarantees instant visibility of variable state updates across multiple active threads.

Comparison: Volatile is a concurrency modifier addressing thread visibility and memory coherency during execution, while transient is a persistence modifier addressing field exclusions during byte-stream transformations. They address completely distinct runtime aspects.

Explain volatile, ThreadLocal, static, final, transient, and Comparable vs Comparator with complete Java examples.

Compare the core Garbage Collectors (G1GC vs ZGC vs CMS) and explain how you detect and fix memory leaks in production.

JVM Garbage Collectors employ different algorithms to reclaim unreachable object memory:

  • CMS (Concurrent Mark Sweep): Deprecated in Java 9, removed in 14. An older, multi-threaded collector that marks and sweeps concurrently with application execution. Suffered from heap fragmentation (no compaction phase), leading to unpredictable Stop-The-World (STW) pauses.
  • G1GC (Garbage First): The default collector since Java 9. The heap is split into equal-sized virtual regions. G1GC targets regions with the most garbage first (hence 'Garbage First') and compacts memory incrementally. It allows configuring target STW pause limits (e.g., `-XX:MaxGCPauseMillis=200`).
  • ZGC (Z Garbage Collector): Production-ready in Java 15. A scalable, ultra-low latency collector. It performs all heavy GC phases concurrently with application threads (marking, compaction, reference processing) using **load barriers** and **colored pointers**. STW pauses are guaranteed to be under 1ms, regardless of heap size (from megabytes to terabytes).

Detecting and Fixing Memory Leaks:

A memory leak in Java occurs when unused objects remain reachable in the GC root chain (e.g., static collections retaining records, unclosed resources, listeners).

  1. Detection Tools: Use APMs (Datadog, Dynatrace), JVM utilities (`jmap`, `jstat`, `jcmd`), or profilers (VisualVM, JProfiler, Eclipse MAT).
  2. Process: Generate a Heap Dump when memory usage is high (`jmap -dump:format=b,file=heap.hprof `). Open it in Eclipse MAT and run a "Leak Suspects" report to find which class loaders or objects dominate Heap memory.
  3. Fixes: Make sure static caches have size limits, remove listeners/callbacks, wrap resources in try-with-resources, and prefer weak/soft references (WeakHashMap) for caching.

Explain the JVM Memory Model. What are Heap, Stack, Metaspace, and the difference between PermGen and Metaspace?

The JVM splits runtime data memory into several distinct areas:

  • Stack Memory: Thread-local execution space. Stored in LIFO order, containing local variables, primitive values, and reference handles to objects on the Heap. Stack memory is allocated per thread; once a method finishes execution, its stack frame is instantly popped and cleared. Thread stacks do not undergo Garbage Collection.
  • Heap Memory: Global shared memory space where all Java objects, arrays, and instance variables are allocated. The Heap is divided into the Young Generation (Eden, Survivor spaces S0/S1) and Old/Tenured Generation. Young Gen holds short-lived allocations (Minor GC), while long-lived objects are promoted to Old Gen (Major/Full GC).
  • Metaspace (Java 8+): Replaced the old **PermGen** (Permanent Generation). It stores class definitions, method metadata, runtime constant pool, and annotations.

Difference: PermGen vs Metaspace
  • Memory Allocation: PermGen was part of the contiguous JVM Heap, with a fixed max size (`-XX:MaxPermSize`). Sizing errors frequently caused `java.lang.OutOfMemoryError: PermGen space`. Metaspace is allocated from **native off-heap memory** (unlimited by default, bounded only by OS physical memory). You can cap it using `-XX:MaxMetaspaceSize`.
  • Garbage Collection: Metaspace undergoes class unloading and garbage collection automatically once class loader instances become unreachable.

Difference between synchronized and ReentrantLock?

synchronized is an intrinsic JVM keyword that provides structured mutual exclusion. The lock is acquired on the object monitor automatically upon entering the block/method and released when exiting (even in exception cases). It cannot be configured for fair locking, nor can it be polled without blocking.

ReentrantLock (from `java.util.concurrent.locks`) is a class-based lock. It must be manually locked and unlocked (within a `try-finally` block). It offers features like:

  • tryLock(): Attempt to acquire the lock immediately or with a timeout. Helps prevent blocking.
  • lockInterruptibly(): Allows waiting threads to respond to interrupts.
  • Fair Locking: Can prioritize threads that have been waiting the longest.

Deadlock Prevention and Resolution:

A deadlock occurs when Thread A holds Lock 1 and waits for Lock 2, while Thread B holds Lock 2 and waits for Lock 1.

  • Lock Ordering: Ensure all threads acquire locks in the exact same order.
  • Lock Timeout: Use `ReentrantLock.tryLock(timeout, unit)` instead of blocking.
  • Diagnostics: Use `jcmd Thread.print` or `jstack ` to extract thread dumps. The JVM automatically identifies deadlocked threads and prints their lock ownership chains.

DeadlockPrevention.java 
import java.util.concurrent.locks.ReentrantLock;
import java.util.concurrent.TimeUnit;

public class DeadlockPrevention {
    private final ReentrantLock lockA = new ReentrantLock();
    private final ReentrantLock lockB = new ReentrantLock();

    public void safeMethod() {
        boolean acquiredA = false;
        boolean acquiredB = false;
        try {
            // Attempt to acquire both locks within 200ms
            acquiredA = lockA.tryLock(200, TimeUnit.MILLISECONDS);
            acquiredB = lockB.tryLock(200, TimeUnit.MILLISECONDS);
            
            if (acquiredA && acquiredB) {
                // Critical section safely executed
                System.out.println("Processing transaction under safe lock ownership.");
            } else {
                System.out.println("Could not acquire locks safely. Backing off to retry...");
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        } finally {
            if (acquiredB) lockB.unlock();
            if (acquiredA) lockA.unlock();
        }
    }
}

What is Idempotence in API design? How do you implement idempotent POST endpoints?

import jakarta.servlet.*;

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        HttpServletResponse httpResponse = (HttpServletResponse) response;

                // Track request key (In production set TTL like 1 hour)

            }
        }
        
        chain.doFilter(request, response);
    }

Design a pluggable, concurrent workflow execution engine. Combine Strategy, Factory, and State patterns to adhere to SOLID principles.

To design a highly extensible, concurrent workflow engine, we map these patterns to distinct responsibilities:

  1. Strategy Pattern: Encapsulates individual step execution algorithms. Adding a new task simply requires implementing a `TaskStrategy` interface.
  2. Factory Pattern: Instantiates the correct task execution strategy dynamically based on metadata definitions.
  3. State Pattern: Manages the lifecycle transitions of the Workflow instance (`PENDING`, `RUNNING`, `COMPLETED`, `FAILED`).
By combining these patterns, we prevent code bloat, isolate concerns, and make the workflow engine fully compliant with the **Open-Closed Principle** (adding new task types without editing core execution engines).

WorkflowEngine.java 
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;

// State Interface
interface WorkflowState {
    void handle(WorkflowContext ctx);
}

class RunningState implements WorkflowState {
    @Override
    public void handle(WorkflowContext ctx) {
        System.out.println("Workflow is running tasks concurrently...");
    }
}

class CompletedState implements WorkflowState {
    @Override
    public void handle(WorkflowContext ctx) {
        System.out.println("Workflow finalized successfully. Cleaning up resources.");
    }
}

class WorkflowContext {
    private WorkflowState state = new RunningState();
    public void setState(WorkflowState state) { this.state = state; }
    public void execute() { state.handle(this); }
}

// Strategy Interface for Tasks
interface TaskStrategy {
    void executeTask(String payload);
}

class EmailTask implements TaskStrategy {
    @Override public void executeTask(String p) { System.out.println("Sending email: " + p); }
}

class DbWriteTask implements TaskStrategy {
    @Override public void executeTask(String p) { System.out.println("Persisting: " + p); }
}

// Factory to resolve Strategy
class TaskFactory {
    private static final Map<String, TaskStrategy> strategies = new HashMap<>();
    static {
        strategies.put("EMAIL", new EmailTask());
        strategies.put("DB", new DbWriteTask());
    }
    public static TaskStrategy get(String type) {
        return strategies.getOrDefault(type, p -> System.out.println("No-Op Task"));
    }
}

// Orchestrator Engine
public class WorkflowEngine {
    private final ExecutorService executor = Executors.newFixedThreadPool(4);

    public void processWorkflow(String taskType, String payload) {
        WorkflowContext ctx = new WorkflowContext();
        ctx.execute(); // Enter Running State

        executor.submit(() -> {
            TaskStrategy strategy = TaskFactory.get(taskType);
            strategy.executeTask(payload);
            
            ctx.setState(new CompletedState());
            ctx.execute(); // Transition to Completed State
        });
    }
    
    public void shutdown() { executor.shutdown(); }
}

Outline Singleton, Factory vs Abstract Factory, Strategy, Builder, Observer, Proxy, Decorator, and Adapter patterns.

These Gang of Four (GoF) patterns form the core of reusable object-oriented architectures:

  • Singleton: Restricts class instantiation to a single instance. Must prevent double-instantiation via multi-threading (using double-checked locking or JVM ClassLoader-based holder/enums).
  • Factory Method vs Abstract Factory: Factory Method uses inheritance to let subclasses decide which product to instantiate. Abstract Factory uses composition to create families of related products without specifying concrete classes.
  • Strategy: Defines a family of interchangeable algorithms and injects them dynamically. Promotes composition over inheritance.
  • Builder: Separates complex object construction from its representation, allowing step-by-step object assembly (essential for immutable structures).
  • Observer: Sets up a one-to-many subscription model so dependent objects receive automatic notifications when state changes.
  • Proxy vs Decorator vs Adapter:
    • **Proxy**: Controls access to the target object (e.g., lazy loading, security, caching, transactional aspects). Has the exact same interface.
    • **Decorator**: Adds responsibilities to an object dynamically without changing its interface.

Explain the self-invocation proxy gotcha in Spring AOP and how to bypass it. Show a custom Aspect for distributed rate limiting.

Spring AOP is proxy-based. When a client bean calls a method on another target bean, Spring routes the call through an intermediate proxy, executing any woven aspects (like `@Transactional` or custom rate limiters). However, if a bean calls one of its own internal methods (self-invocation), the call bypasses the proxy entirely, resolving to direct JVM `this` reference calls. As a result, **no aspect logic executes**.

Bypassing Self-Invocation:
  1. Self-Injection (Recommended): Inject the bean into itself using lazy autowiring. Internal calls are made on the self-injected variable, routing through the proxy.
  2. AopContext.currentProxy(): Access the proxy directly via `((MyService) AopContext.currentProxy()).internalMethod()`. Requires setting `@EnableAspectJAutoProxy(exposeProxy = true)`.
  3. Refactoring: Move the self-invoked method to a separate helper helper service bean.

import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.Pointcut;
import org.springframework.context.annotation.Lazy;
import org.springframework.stereotype.Component;
import java.util.concurrent.ConcurrentHashMap;

@Aspect
@Component
public class RateLimiterAspect {
    private final ConcurrentHashMap<String, Long> clientRequestTimestamps = new ConcurrentHashMap<>();
    private final MyService self;

    // Self-injection to bypass self-invocation gotcha if needed!
    public RateLimiterAspect(@Lazy MyService self) {
        this.self = self;
    }

    @Pointcut("@annotation(RateLimit)")
    public void rateLimitedMethod() {}

    @Around("rateLimitedMethod()")
    public Object enforceLimit(ProceedingJoinPoint joinPoint) throws Throwable {
        // Mocking client tracking (In production, retrieve IP/JWT token from SecurityContext)
        String clientIp = "127.0.0.1"; 
        
        long currentTime = System.currentTimeMillis();
        long lastRequest = clientRequestTimestamps.getOrDefault(clientIp, 0L);

        if (currentTime - lastRequest < 1000) { // Limit: 1 request per second
            throw new RuntimeException("HTTP 429: Too Many Requests. Rate limit exceeded.");
        }

        clientRequestTimestamps.put(clientIp, currentTime);
        return joinPoint.proceed(); // Safe execution
    }
}

What Is Aspect-Oriented Programming?

AOP (Aspect-Oriented Programming) complements OOP by allowing developers to modularize cross-cutting concerns (e.g., logging, transaction management, caching, security) that slice across multiple business classes.

Key AOP Terminologies:
  • Aspect: The module containing the cross-cutting concern (e.g., a `TransactionAspect` class). Contains Pointcuts and Advices.
  • JoinPoint: A point during execution of the program (such as method execution or exception handling) where an aspect can be plugged in. In Spring AOP, a JoinPoint is **always method execution**.
  • Pointcut: A predicate expression matching JoinPoints. It defines *where* the advice should run (e.g., execution of any method in `com.service.*`).
  • Advice: Action taken by an aspect at a particular JoinPoint. Types: `@Before`, `@After`, `@AfterReturning`, `@AfterThrowing`, and `@Around` (most powerful: can block execution, modify return values, and handle exceptions).
Types of Weaving (Plugging aspects into application types):
  1. Compile-Time Weaving: Aspects are woven into Java bytecode at compilation using the AspectJ compiler (`ajc`). Has zero runtime performance overhead.
  2. Post-Compile Weaving (Binary Weaving): Woven into existing compiled `.class` or `.jar` files.
  3. Load-Time Weaving (LTW): Woven dynamically when classes are loaded into JVM memory by using a JVM Java Agent (`-javaagent:aspectjweaver.jar`).
  4. Runtime Weaving (Spring AOP Default): Uses dynamic proxies (JDK dynamic proxies for interfaces, CGLIB subclasses for concrete classes) to intercept method calls at execution time. Limited to public method calls initiated outside the bean.

What Is Weaving?

Weaving is the process of linking aspects with other application types or objects to create an advised object. It can occur at: (1) Compile-time (using AspectJ compiler); (2) Class-load time (using dynamic classloaders); or (3) Runtime (Spring's default, using JDK dynamic proxies or CGLIB).

import jakarta.servlet.*;
import jakarta.servlet.http.HttpServletRequest;
import org.slf4j.MDC;
import org.springframework.stereotype.Component;
import java.io.IOException;
import java.util.UUID;

@Component
public class LoggingFilter implements Filter {
    private static final String TRACE_ID_KEY = "traceId";

    @Override
    public void doFilter(ServletRequest request, ServletResponse response, FilterChain chain)
            throws IOException, ServletException {
        HttpServletRequest httpRequest = (HttpServletRequest) request;
        
        // Extract traceId from incoming headers, or generate one if absent
        String traceId = httpRequest.getHeader("X-Correlation-ID");
        if (traceId == null || traceId.isEmpty()) {
            traceId = UUID.randomUUID().toString();
        }

        // Bind traceId to thread-local MDC context
        MDC.put(TRACE_ID_KEY, traceId);

        try {
            chain.doFilter(request, response);
        } finally {
            MDC.remove(TRACE_ID_KEY); // Clean context to prevent thread leak
        }
    }
}

How does Spring Boot auto-configuration and starters work internally? Provide a complete custom starter auto-configuration.

Spring Boot Starters are dependency aggregators. Adding `spring-boot-starter-web` imports Tomcat, Jackson, validation, and Spring MVC libraries under a single dependency block, ensuring version compatibility.

Auto-Configuration Internals: At startup, Spring Boot scans files named `META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` (Spring Boot 3.x) or `META-INF/spring.factories` (older versions) located inside starter jars on the classpath.   • @ConditionalOnClass: Activate only if specific classes are present on the classpath.

  • @ConditionalOnProperty: Activate only if a configuration property matches expected values.

How do you stream data non-blockingly using WebClient and CompletableFuture in Spring WebFlux?

Reactive Streams: WebFlux handles streaming using reactive types (Mono, Flux) mapped to the text/event-stream media type.

Producers: Stream items asynchronously using reactive pipelines, preventing worker thread blocks.

Receivers: WebClient consumes streams non-blockingly using bodyToFlux(Data.class), processing elements as they arrive.

CompletableFuture Bridge: To bridge blocking CompletableFuture logic, use Mono.fromFuture() or Flux.create() to adapt async operations into reactive streams.

import org.springframework.web.bind.annotation.GetMapping;
import org.springframework.web.bind.annotation.RestController;
import org.springframework.web.reactive.function.client.WebClient;
import reactor.core.publisher.Flux;
import reactor.core.publisher.Mono;
import java.time.Duration;
import java.util.concurrent.CompletableFuture;

@RestController
public class ReactiveStreamController {

    // 1. Producer Endpoint - Streams data as Server-Sent Events (SSE)
    @GetMapping(value = "/stream/events", produces = MediaType.TEXT_EVENT_STREAM_VALUE)
    public Flux<String> streamEvents() {
        return Flux.interval(Duration.ofSeconds(1))
                   .map(seq -> "Event Sequence: " + seq);
    }

    // 2. Bridge async Future to Reactive Mono
    public Mono<String> bridgeFuture() {
        CompletableFuture<String> future = CompletableFuture.supplyAsync(() -> "Async Data");
        return Mono.fromFuture(future); // Safe non-blocking bridge
    }

    // 3. Receiver Service - Consumes SSE stream using WebClient
    public void consumeStream() {

Explain Reactive Streams backpressure. How does Spring WebFlux handle it, and what are Mono and Flux?

Backpressure is the mechanism in Reactive Streams that allows a **Consumer** to signal to a **Publisher** how much data it can process at any given moment. This prevents a fast publisher from overwhelming a slow consumer, avoiding memory exhaustion and out-of-memory errors.

In WebFlux, Reactor handles backpressure automatically through TCP flow control. If Netty's outbound write buffers are full, the consumer slows down its demand requests (`Subscription.request(n)`), signaling the upstream publisher to slow down or buffer data.

Mono vs Flux
  • Mono<T>: A Reactive Streams publisher that emits **0 or 1** element, then completes. Useful for single entities, updates, or void results.
  • Flux<T>: A Reactive Streams publisher that emits **0 to N** elements, then completes. Useful for streaming datasets, real-time events, or chunked outputs.

How do you handle blocking database calls (e.g. JDBC) inside a Spring WebFlux application without starving the Netty Event Loop?

Spring WebFlux is built on a non-blocking reactive event-loop execution model (typically running on Netty). The Netty event loop runs on a very small number of threads (matching CPU cores). If a thread blocks inside the event loop (e.g., executing a synchronous JDBC query, calling a slow blocking REST API, or doing heavy cryptography), **the entire application freezes** because no other incoming requests can be polled.

Scheduler-Bypass Strategy:

To safely invoke blocking library code, isolate the execution using publishOn() to offload execution onto a dedicated thread pool (Schedulers.boundedElastic()). This keeps the main Netty Event Loop threads free to process non-blocking network I/O.

import reactor.core.scheduler.Schedulers;
import org.springframework.stereotype.Service;

        return Mono.fromCallable(() -> fetchFromBlockingDb(id))
            // Offload the blocking DB call to boundedElastic scheduler pool
            .subscribeOn(Schedulers.boundedElastic()) 
            .map(data -> data + " | Processed on Non-blocking Pipeline");

If millions of data transactions need processing via a REST API, what are the best design practices?

Processing millions of records synchronously in a single HTTP request is an anti-pattern. It leads to timeouts, high memory usage, and blocks the client. Here are the correct production-grade approaches:

  1. Asynchronous Processing & Job Status Polling: The API accepts the payload, validates the schema, publishes the job to a message queue or database, and returns `202 Accepted` immediately with a status tracking URL. The client polls this URL (`/api/v1/jobs/{jobId}`) to check progress.
  2. Chunked Batch Processing (Spring Batch): Break the large load into manageable chunks (e.g., 500-1000 records). Read chunk -> Process/Validate -> Write. Use database-level bulk operations (like `EntityManager.flush()` and `clear()` to prevent memory exhaustion).
  3. Streaming with Kafka: Instead of a single large payload, stream records individually to a Kafka topic. A group of consumer services processes these records asynchronously at their own rate, providing natural backpressure and scaling.

BulkProcessingPattern.java 
import jakarta.persistence.EntityManager;
import org.springframework.transaction.annotation.Transactional;
import java.util.List;

public class BulkProcessingPattern {
    private final EntityManager entityManager;

    public BulkProcessingPattern(EntityManager em) { this.entityManager = em; }

    @Transactional
    public void executeBatchInsert(List<Object> largeCollection) {
        int batchSize = 500;
        for (int i = 0; i < largeCollection.size(); i++) {
            entityManager.persist(largeCollection.get(i));
            
            // Periodically flush updates to the DB and clear memory
            if (i > 0 && i % batchSize == 0) {
                entityManager.flush(); // Sends sql updates to database
                entityManager.clear(); // Evicts entity cache to prevent OutOfMemoryError
            }
        }
    }
}

How do you handle distributed payment processing idempotency safely in a Spring REST Controller? Provide a thread-safe Redis-based solution.

Idempotency ensures that making the same API request multiple times produces the exact same outcome as a single request, preventing duplicate charges or duplicate database records. Clients must submit a unique header like `Idempotency-Key` (typically a UUID).

Our solution utilizes Redis to create an atomic **distributed lock** with a Time-To-Live (TTL). When a request arrives:

  1. Check if the key exists in Redis. If yes, reject the request as duplicate or return the cached response.
  2. If no, use Redis `SETNX` (Set if Not Exists) to atomically acquire the lock with a TTL (e.g., 10 minutes) and process the payment.
  3. Store the final transaction response in Redis under the idempotency key, allowing subsequent duplicate requests to return the cached response immediately.

IdempotencyInterceptor.java 
import org.springframework.data.redis.core.StringRedisTemplate;
import org.springframework.stereotype.Component;
import org.springframework.web.servlet.HandlerInterceptor;
import jakarta.servlet.http.HttpServletRequest;
import jakarta.servlet.http.HttpServletResponse;
import java.time.Duration;

@Component
public class IdempotencyInterceptor implements HandlerInterceptor {

    private final StringRedisTemplate redisTemplate;

    public IdempotencyInterceptor(StringRedisTemplate redisTemplate) {
        this.redisTemplate = redisTemplate;
    }

    @Override
    public boolean preHandle(HttpServletRequest req, HttpServletResponse res, Object handler) throws Exception {
        String key = req.getHeader("Idempotency-Key");
        if (key == null || key.isBlank()) {
            res.sendError(400, "Missing Idempotency-Key header.");
            return false;
        }

        // Redis SETNX: Set key only if it doesn't exist, with 10-minute expiration
        String redisKey = "idempotency:" + key;
        Boolean isLockAcquired = redisTemplate.opsForValue()
            .setIfAbsent(redisKey, "PROCESSING", Duration.ofMinutes(10));

        if (Boolean.FALSE.equals(isLockAcquired)) {
            String status = redisTemplate.opsForValue().get(redisKey);
            if ("PROCESSING".equals(status)) {
                res.sendError(409, "Concurrent duplicate request in progress. Retry shortly.");
            } else {
                res.setStatus(200); // Return cached output
                res.getWriter().write(status); 
            }
            return false;
        }
        return true; // Lock acquired safely, proceed to Controller
    }
}

Explain the Circuit Breaker pattern with a real microservice example. How do you declare fallback methods in Spring Boot?

Circuit Breaker Pattern: Resilience4j intercepts calls to external services. If failures exceed a set threshold, the breaker transitions to the OPEN state.

Fallback Execution: In the OPEN state, requests fail-fast to local fallback methods, preventing cascading failures across the system.

Half-Open Testing: The breaker enters HALF-OPEN after a cooldown period, executing test calls to determine if the downstream service has recovered.

How do you optimize Spring Boot for serverless (AWS Lambda) using GraalVM? Detail Native Image optimization and Runtime Hints.

Classic Spring Boot applications have a high startup time (warm-up) and a large memory footprint, making them inefficient for serverless environments where cold starts incur latency and cost.

GraalVM Native Image compiles Java bytecode ahead-of-time (AOT) into a standalone native executable tailored for a specific target OS. This results in **millisecond-level startup times** and a drastically reduced memory footprint, making it ideal for AWS Lambda.

The Challenge: Reflection & Dynamic Proxies: GraalVM performs closed-world analysis, assuming all reachable classes and resources must be known at compile-time. Dynamic features like Java Reflection, resource bundle loading, serialization, and dynamic proxies (extensively used in Spring) are not automatically visible to the compiler and will fail at runtime.

To resolve this, we configure RuntimeHintsRegistrar to declare dynamic classes, resource paths, and reflection bindings, allowing GraalVM to compile them into the binary safely.

LambdaRuntimeHint.java 
import org.springframework.aot.hint.RuntimeHints;
import org.springframework.aot.hint.RuntimeHintsRegistrar;
import org.springframework.aot.hint.TypeReference;

// Declare reflection runtime hint requirements for GraalVM AOT Compilation
public class LambdaRuntimeHint implements RuntimeHintsRegistrar {
    @Override
    public void registerHints(RuntimeHints hints, ClassLoader classLoader) {
        // Register custom DTO for reflection mapping (e.g., Jackson JSON serialization)
        hints.reflection().registerType(
            TypeReference.of("com.serverless.dto.UserRequest"),
            builder -> builder.withMembers(
                org.springframework.aot.hint.MemberCategory.INVOKE_PUBLIC_CONSTRUCTORS,
                org.springframework.aot.hint.MemberCategory.INVOKE_PUBLIC_METHODS
            )
        );

        // Register classpath resource bundle
        hints.resources().registerPattern("configs/custom-schema.json");
    }
}

Compare Saga Orchestration vs Choreography. Explain how the Outbox Pattern ensures eventual consistency.

In a microservice architecture, managing transactions across multiple databases requires eventual consistency strategies:

  • Saga Pattern: Decomposes a distributed transaction into a sequence of local database transactions. If one step fails, the Saga triggers compensatory transactions (rollback steps) in reverse order.
    • **Choreography**: Services publish and subscribe to events without a central coordinator. High decoupling, but difficult to track flow complexity.
    • **Orchestration**: A central coordinator service directs the workflow, telling each service when to execute its transaction. Simpler monitoring, but introduces a single point of failure and coupling.
  • Transactional Outbox Pattern: Prevents data inconsistency when updating a database and publishing message events (like Kafka) in the same action. Writing directly to the message broker can fail due to network drops, leaving the DB updated but the message lost. Instead, write both the domain entity update and an outbox event into the **same database transaction** (in an `OUTBOX` table). An independent background poll process (or a Change Data Capture tool like Debezium) reads the `OUTBOX` table and reliably publishes messages to the broker.

How do you implement retry, fallback, and circuit breaker in microservices? Provide a Resilience4j code example.

In distributed systems, failures of downstream microservices are inevitable. To prevent a cascade of failures, apply resilience patterns:

  • Circuit Breaker: Monitors error rates on calls. If errors exceed a threshold (e.g., 50% failures), the circuit **trips open**, failing subsequent calls instantly without hitting the downstream service. After a cooldown period, it enters a **half-open** state to check if the downstream service has recovered.
  • Retry: Retries failed calls with exponential backoff and jitter to handle transient network blips.
  • Fallback: Provides a default, degraded response (e.g., cached data or static catalog) when the downstream call fails or is short-circuited.

import io.github.resilience4j.circuitbreaker.annotation.CircuitBreaker;
import io.github.resilience4j.retry.annotation.Retry;

import java.util.Collections;
import java.util.List;

@Service
public class ProductService {
    private final RestTemplate restTemplate;

    public ProductService(RestTemplate restTemplate) {
        this.restTemplate = restTemplate;
    }

    @CircuitBreaker(name = "catalogService", fallbackMethod = "fallbackProducts")
    @Retry(name = "catalogService")
    public List<String> getProducts() {
        // Direct blocking API call to downstream microservice

    }

    // Fallback method must have matching signature + exception parameter
    public List<String> fallbackProducts(Throwable exception) {

        return Collections.singletonList("Static Catalog Item (Fallback)");
    }
}

Explain Kafka Producer, Consumer, Offset, and Group Rebalancing. How do you handle message spikes?

Producer & Consumer: Producers write records to partition keys, which determines the target partition log. Consumers subscribe to topics, reading records sequentially.

Offsets: Offsets record the sequential read progress of each consumer, which is stored in the internal `__consumer_offsets` topic.

Consumer Groups: Multiple consumers in a consumer group read from separate partitions in parallel, which is managed by a coordinator.

Handling Spikes: Scale partitions to distribute load. Designing topics: set partitions >= consumer count, use specific partition keys to prevent hot spots, and configure replication factors.

Write a robust, non-blocking Kafka consumer with Spring Boot that handles processing failures using retry topics and routes poison pills to a DLT.

In high-throughput event processing, a single malformed message (poison pill) or a transient processing failure must not halt the consumer. Blocking retries (`Thread.sleep()`) are problematic because if processing takes too long, the broker assumes the consumer is dead, triggering partition rebalancing and halting the pipeline.

The standard solution is **Non-blocking Retries & Dead Letter Topics (DLT)**:

  1. If a message fails, catch the exception and immediately publish the record to a retry topic (e.g., `orders-retry-10s`) and commit the offset on the main topic.
  2. An independent consumer processes messages from the retry topic after a delay.
  3. If retries are exhausted, the recovery coordinator publishes the message to a Dead Letter Topic (`orders-dlt`) for manual debugging, allowing the main pipeline to continue processing.

What are the core components of Kafka? Explain partition structuring, consumer groups, offset tracking, and ZooKeeper vs KRaft.

Apache Kafka is a distributed event streaming platform built on an append-only transaction log:

  • Brokers & Clusters: Brokers are nodes that receive, write, and serve data records. A cluster is a group of brokers cooperating for high availability.
  • Topics & Partitions: A Topic is a logical category of events. Topics are split into **Partitions** (physical commit logs). Partitions allow Kafka to scale by distributing data across multiple brokers, enabling concurrent writes and reads.
  • Consumer Groups & Rebalancing: A group of consumers that cooperate to read messages from a set of partitions. Each partition is assigned to **exactly one** consumer in the group, ensuring message ordering per partition. If a consumer joins or leaves, Kafka triggers a **rebalance** to reassign partitions.
  • Offset Tracking: Consumers track their reading progress using offsets. Offsets are committed back to a special internal Kafka topic: `__consumer_offsets`.
  • ZooKeeper vs KRaft: Historically, Kafka used ZooKeeper to manage cluster metadata, leader elections, and configuration. In modern Kafka, **KRaft** (Kafka Raft Metadata Mode) replaces ZooKeeper. KRaft stores metadata directly inside Kafka partitions managed by controller nodes, simplifying architecture, speeding up partition leader elections, and improving scalability.

What are SQL ACID properties? Compare Stored Procedures vs Functions in relational database design.

ACID Properties: Atomicity (all operations commit or all rollback), Consistency (data state complies with schema constraints), Isolation (concurrent transactions execute independently), and Durability (committed changes survive system failures).

Stored Procedures: Precompiled blocks of SQL statements executed on the database server. They can execute transactional commits/rollbacks, return multiple result sets, and perform raw DML operations.

Functions: Must return exactly one value. They cannot execute transactional commands or modify database state, but can be embedded inside queries.

Best Practice: Use stored procedures for complex maintenance jobs and batch data operations, and functions for calculated expressions (like custom string formatting).

How do you maintain database consistency under high concurrency? Explain Optimistic vs Pessimistic locking.

When multiple concurrent API calls update the same database records simultaneously, conflicts occur (lost updates, dirty reads). We manage concurrency using locking mechanisms:

  • Optimistic Locking: Assumes conflicts are rare. Every record includes a `@Version` attribute (integer or timestamp). When updating, the SQL includes a check: `UPDATE table SET ... version = version + 1 WHERE id = ? AND version = :currentVersion`. If another transaction updated the record first, the version mismatch causes the update to fail, throwing an OptimisticLockingFailureException. This approach is non-blocking and highly scalable.
  • Pessimistic Locking: Assumes conflicts are frequent. It locks rows physically at the database level using `SELECT ... FOR UPDATE` (Pessimistic Write). Concurrent readers/writers must block and wait until the transaction completes and releases the lock. This prevents conflicts but reduces concurrency and can lead to database deadlocks.

import jakarta.persistence.*;

    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private double balance;

    private int version;

    public void withdraw(double amount) {
        if (balance < amount) throw new RuntimeException("Insufficient funds.");
        this.balance -= amount;
    }
}

What is the JPA N+1 problem and how do you diagnose and fix it? Provide a JPA repository JOIN FETCH solution.

The N+1 problem occurs when Hibernate loads a parent entity, then issues N additional SQL queries to load child associations for each of the N parent entities (typically caused by lazy loading or using standard `findAll()`).

For example, if you fetch a list of 100 `Author` records and iterate over them to print their `Book` objects:

  • 1 query is executed to fetch the 100 `Author` objects.
  • 100 additional queries are executed to fetch the `Book` objects for each author individually.
This results in 101 database roundtrips, degrading application performance.

Resolution (JOIN FETCH):

Instruct JPA to load both parent and child associations in a single database roundtrip using a SQL **INNER JOIN FETCH** or **LEFT JOIN FETCH** in a custom JPQL query.

Compare SQL Joins (INNER, LEFT, RIGHT, FULL) and Indexing internals (B-Tree, Clustered vs Non-Clustered).

Understanding database retrieval mechanics is essential for query optimization:

  • SQL Joins:
    • **INNER JOIN**: Returns rows only when there is a match in both tables.
    • **LEFT JOIN**: Returns all rows from the left table, plus matched rows from the right table. Unmatched right-side values yield `NULL`.
    • **RIGHT JOIN**: Returns all rows from the right table, plus matched rows from the left table.
    • **FULL JOIN**: Returns all rows when there is a match in either the left or right table.
  • Indexing Internals (B-Tree): An index is a balanced tree data structure (B-Tree / B+ Tree) that avoids O(N) table scans, enabling O(log N) searches. Nodes contain keys and child pointers, guiding queries down to the correct data pages.
  • Clustered vs Non-Clustered Indexes:
    • **Clustered Index**: Determines the physical storage order of rows in the table. There can be **exactly one** clustered index per table (typically the Primary Key). The leaf nodes of a clustered index contain the actual data rows.
    • **Non-Clustered Index**: A separate structure containing index keys and pointers (row locators/logical addresses) back to the actual data rows in the clustered index. A table can have multiple non-clustered indexes.

Algorithms & Data Structures

Implement in-place LinkedList reversal, Binary Search, and explain BFS vs DFS graph traversals.

Key algorithmic patterns frequently used in software development:

  • In-Place LinkedList Reversal: Reverse a singly linked list iteratively by maintaining three pointers: `prev`, `curr`, and `next`. Point `curr.next` to `prev`, then move the pointers forward. O(N) time complexity, O(1) auxiliary space complexity.
  • Binary Search: A divide-and-conquer search algorithm on a **sorted array**. Compare the search key with the array's midpoint. If they match, return the index. If the key is smaller, narrow search to the left half; if larger, to the right half. O(log N) search time complexity.
  • BFS vs DFS Graph Traversals:
    • **BFS (Breadth-First Search)**: Explores a graph level-by-level, visiting all neighbor nodes first. Implemented using a **Queue**. Best for finding the shortest path in unweighted graphs.
    • **DFS (Depth-First Search)**: Explores a branch as deep as possible before backtracking. Implemented using recursion or a **Stack**. Best for topological sorting, finding connected components, and resolving dependencies.

public class LinkedListInPlaceReversal {
    static class Node {
        int val;
        Node next;
        Node(int val) { this.val = val; }
    }

    public static Node reverse(Node head) {
        Node prev = null;
        Node curr = head;
        Node next = null;

        while (curr != null) {
            next = curr.next; // Store next node
            curr.next = prev; // Reverse current node link
            prev = curr;      // Move prev pointer forward
            curr = next;      // Move curr pointer forward
        }
        return prev; // New head of reversed list
    }
}

Implement a thread-safe, high-performance Least Recently Used (LRU) cache with a Time-To-Live (TTL) expiration mechanism in pure Java.

A classic LRU cache maintains O(1) reads and writes by combining a **HashMap** (for lookup) with a **Doubly Linked List** (to track usage order). When a key is accessed, we move its node to the head of the list. When the cache is full, we evict the node at the tail of the list.

To support a Time-To-Live (TTL) expiration mechanism, we store the creation/write timestamp in each cache node. During read access, we check if the current time exceeds the node's TTL. If expired, we remove the node immediately and return null. We ensure thread safety by wrapping accesses inside a ReentrantReadWriteLock, permitting multiple concurrent readers but isolating writers.

ConcurrentLruCache.java 
import java.util.HashMap;
import java.util.Map;
import java.util.concurrent.locks.ReentrantReadWriteLock;

public class ConcurrentLruCache<K, V> {
    private static class CacheNode<K, V> {
        K key;
        V value;
        long expiryTime;
        CacheNode<K, V> prev, next;
        
        CacheNode(K key, V value, long ttlMs) {
            this.key = key;
            this.value = value;
            this.expiryTime = System.currentTimeMillis() + ttlMs;
        }
    }

    private final int capacity;
    private final Map<K, CacheNode<K, V>> map = new HashMap<>();
    private final CacheNode<K, V> head, tail;
    private final ReentrantReadWriteLock lock = new ReentrantReadWriteLock();

    public ConcurrentLruCache(int capacity) {
        this.capacity = capacity;
        this.head = new CacheNode<>(null, null, 0);
        this.tail = new CacheNode<>(null, null, 0);
        head.next = tail;
        tail.prev = head;
    }

    public V get(K key) {
        lock.readLock().lock();
        try {
            CacheNode<K, V> node = map.get(key);
            if (node == null) return null;
            
            // Check TTL expiration
            if (System.currentTimeMillis() > node.expiryTime) {
                // Upgrade to write lock to evict expired item
                lock.readLock().unlock();
                lock.writeLock().lock();
                try {
                    evictNode(node);
                } finally {
                    lock.writeLock().unlock();
                    lock.readLock().lock(); // Downgrade lock
                }
                return null;
            }
            
            // Move accessed node to head (indicates recently used)
            lock.readLock().unlock();
            lock.writeLock().lock();
            try {
                moveToHead(node);
            } finally {
                lock.writeLock().unlock();
                lock.readLock().lock();
            }
            return node.value;
        } finally {
            lock.readLock().unlock();
        }
    }

    public void put(K key, V value, long ttlMs) {
        lock.writeLock().lock();
        try {
            CacheNode<K, V> existing = map.get(key);
            if (existing != null) {
                existing.value = value;
                existing.expiryTime = System.currentTimeMillis() + ttlMs;
                moveToHead(existing);
                return;
            }

            if (map.size() >= capacity) {
                // Evict least recently used (tail.prev)
                evictNode(tail.prev);
            }

            CacheNode<K, V> newNode = new CacheNode<>(key, value, ttlMs);
            map.put(key, newNode);
            addNodeAtHead(newNode);
        } finally {
            lock.writeLock().unlock();
        }
    }

    private void addNodeAtHead(CacheNode<K, V> node) {
        node.next = head.next;
        node.prev = head;
        head.next.prev = node;
        head.next = node;
    }

    private void removeNode(CacheNode<K, V> node) {
        node.prev.next = node.next;
        node.next.prev = node.prev;
    }

    private void moveToHead(CacheNode<K, V> node) {
        removeNode(node);
        addNodeAtHead(node);
    }

    private void evictNode(CacheNode<K, V> node) {
        removeNode(node);
        map.remove(node.key);
    }
}

Explain HashMap internals, bucket indexing, collision resolution, and Java 8 treeification.

HashMap is built on an array of nodes (buckets): `Node[] table`.

Core Operations:
  1. Bucket Indexing: When you call `put(key, value)`, HashMap calculates the key's hash using `hash = key.hashCode() ^ (hash >>> 16)` (spreads high bits down to reduce collisions). The bucket index is calculated using bitwise AND: `index = hash & (n - 1)` (where `n` is the array length, which is always a power of 2).
  2. Collision Resolution: If two keys map to the same index, they are stored in a linked list at that bucket node (chaining).
  3. Java 8 Treeification: If a linked list at a single bucket grows beyond a threshold (TREEIFY_THRESHOLD = 8) and the total table capacity is at least 64, HashMap converts the linked list into a balanced **Red-Black Tree**. This improves search time from O(N) to O(log N) for collided buckets. If the size shrinks below 6, it untreeifies back to a list.

How do you optimize rendering performance in React? Implement an optimized state consumer component with Zustand.

React components re-render whenever their parent re-renders or their state changes. In large applications, unnecessary rendering cycles cause UI lag.

Rendering Optimization Hooks:
  • useMemo: Memoizes expensive computations, executing them only when dependencies change.
  • useCallback: Memoizes function instances, preventing child components from re-rendering due to inline function reference changes.
  • React.memo: A higher-order component that skips rendering a component if its incoming props are unchanged.
State Management: Zustand vs Context API

The Context API causes **all consuming components to re-render** whenever the context value changes, even if a component only consumes a subset of the value. Zustand is a lightweight state management library that solves this. It uses a publish-subscribe model with **selector-based state subscription**, ensuring components re-render only when the specific sliced attribute they select changes.

Explain the React component lifecycle, virtual DOM reconciliation, and the usage rules of Hooks.

React is built on a declarative programming model with efficient UI updates:

  • Virtual DOM Reconciliation (Fiber): React maintains a lightweight virtual DOM tree in memory. When state changes, React generates a new Virtual DOM tree. The **reconciliation algorithm (Fiber)** compares the new tree with the old tree (diffing) and batches the minimal set of updates required to modify the real DOM, avoiding expensive layout reflows.
  • Rules of Hooks: Hooks (introduced in React 16.8) allow functional components to manage state and lifecycle side effects.
    1. **Only Call Hooks at the Top Level**: Do not call hooks inside loops, conditions, or nested functions. This ensures React executes hooks in the exact same order on every render.
    2. **Only Call Hooks from React Functions**: Call hooks only from React functional components or custom hooks.
  • Core Hooks:
    • `useState`: Manages local component state.
    • `useEffect`: Manages side effects (API calls, subscriptions, DOM mutations). Fires asynchronously after rendering completes.
    • `useRef`: Retains mutable values that do not trigger re-renders, and offers direct access to DOM nodes.
